# 构建你自己的 AI 智能体：设计空间指南

> 本文不是写代码的教程，而是一份设计决策清单：在构建生产级 AI 智能体系统时，你必须做出的那些选择——所有素材都来自对 Claude Code 的架构分析。

每个生产级编码智能体都要回答同样的几个设计问题。Claude Code 给出的是其中一套答案。本指南把整个决策空间摊开，让你在充分了解的基础上做出自己的选择。

---

## 决策 1：推理放在哪里？

**问题：** 多少决策逻辑放在模型里，多少放在 harness 代码里？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **最小脚手架** | Claude Code（约 1.6% AI 逻辑） | 模型拥有最大自由度；harness 强制执行边界。押注模型能力随时间不断提升。 |
| **显式状态图** | LangGraph | 开发者控制流程；更容易调试和预测。但会约束模型，需要随能力提升而更新。 |
| **重型规划脚手架** | Devin | 多步规划器 + 任务跟踪器。处理复杂工作流时更可靠，但脚手架本身会成为维护负担。 |

**来自 Claude Code 的关键洞察：** 当前沿模型的能力趋同（SWE-bench 前三名成绩差距在 1% 以内）时，**真正拉开差距的是运行层面的 harness**，而不是模型或脚手架。在确定性基础设施（上下文管理、安全、恢复）上的投入，可能比叠加规划约束带来更高的可靠性。

**可以问自己的问题：**
- 你目标模型的能力如何？能力更强的模型需要更少的脚手架。
- 你的工作流需要多可预测？受监管领域可能需要显式状态图。
- 模型能力提升得有多快？一旦模型能力超出脚手架的假设，重型脚手架就会变成技术债。

## 决策 2：你的安全姿态是什么？

**问题：** 如何防止智能体做出有害的行为？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **拒绝优先与分层强制** | Claude Code（7 个独立层） | 非常安全，但可能产生批准疲劳（93% 的提示在未经审阅的情况下被批准）。需要渐进式信任机制。 |
| **容器隔离** | SWE-Agent、OpenHands（Docker） | 强边界，但粒度粗。容器内的一切操作都被允许；容器外的一切都不可达。 |
| **VCS 回滚** | Aider（基于 git） | 轻量，但只兜底文件变更。网络请求、数据外传、shell 副作用都不在防护范围内。 |
| **仅批准** | 基础聊天机器人 | 简单，但用户规模一大就不再可靠——用户会停止阅读提示。 |

**来自 Claude Code 的关键洞察：** 深度防御只有在各层具有**独立故障模式**时才有效。Claude Code 的各层共享同一种经济约束（token 成本）——一旦命令中子命令超过 50 个，就会彻底跳过安全分析。**设计安全层时，要让它们能够独立失败**。

**关于批准疲劳的关键洞察：** 用户会批准 93% 的权限提示。解决方案不是增加警告，而是**重新划分边界**——通过沙箱和分类器为自主操作创造安全区。

**可以问自己的问题：**
- 你的智能体可能做出的最糟糕的事情是什么？（删除生产数据？发送邮件？泄露代码？）
- 你能通过沙箱化减少用户必须做的决策数量吗？
- 你的安全层之间是否共享故障模式（例如，都依赖 token 预算）？

## 决策 3：你如何管理上下文？

**问题：** 上下文窗口是有限的。你如何决定模型看到什么？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **渐进式压缩管道** | Claude Code（5 层） | 能在尽可能长的时间里保留尽可能多的信息。实现和调试都较复杂。压缩对用户不可见。 |
| **简单截断** | 许多基础智能体 | 易于实现。但可能丢掉关键的早期上下文。 |
| **滑动窗口** | 一些聊天应用 | 行为可预测。但无法在语义层面识别哪些内容才重要。 |
| **RAG（检索增强生成）** | 一些 IDE 集成 | 可以访问整个代码库。但检索质量是瓶颈，检索回来的片段常常缺少前后文。 |
| **一次性摘要** | 一些智能体 | 只做一遍摘要。但单次压缩容易丢掉关键细节。 |

**来自 Claude Code 的关键洞察：** 上下文是塑造几乎所有其他架构决策的**最根本的约束**。惰性加载、工具 schema 延迟加载、子智能体仅回传摘要、为每个工具结果设预算——这些机制之所以存在，都是被上下文稀缺逼出来的。从第一天起就要为上下文稀缺而设计。

**渐进式方法：** 先应用破坏性最小的压缩。预算削减（开销低）→ 历史裁剪（开销低）→ 缓存感知压缩（中等）→ 上下文折叠（中等）→ 完整摘要（开销高，最后手段）。

**可以问自己的问题：**
- 你的上下文窗口有多大？这决定了你需要多激进的压缩。
- 你需要支持长会话（数小时的工作）吗？一次性截断撑不住。
- 你能把"指导"上下文（指令）和"工作"上下文（对话）分开吗？

## 决策 4：你如何处理可扩展性？

**问题：** 外部工具、自定义指令和用户定制如何插入你的系统？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **渐进式上下文成本机制** | Claude Code（hooks=0、skills=低、plugins=中、MCP=高） | 不同扩展在不同成本上运行。管理复杂，但可扩展。 |
| **单一统一 API** | 许多工具使用框架 | 概念简单，易于理解。但每个扩展都消耗上下文，限制可扩展性。 |
| **插件市场** | IDE 扩展 | 有丰富的生态潜力。但质量控制和安全审查会成为瓶颈。 |

**来自 Claude Code 的关键洞察：** 并非所有扩展都必须消耗上下文 token。Hooks（零开销）处理生命周期事件而不触及上下文窗口；Skills（低开销）仅在相关时才注入。把高上下文开销的机制（MCP）专门留给真正要引入一整类新工具的场景。

**三个注入点：** 每个智能体循环都有三处可以干预的位置：
1. **assemble()** —— 模型看到的内容（指令、工具 schema）
2. **model()** —— 模型能触及的内容（可用工具）
3. **execute()** —— 操作是否/如何运行（权限门控、前置/后置钩子）

**可以问自己的问题：**
- 你的智能体需要支持多少工具？工具越多，上下文压力越大。
- 你需要第三方扩展吗？要为安全与质量控制做好规划。
- 你能把工具 schema 的加载延迟到模型真正需要工具时再做吗？

## 决策 5：子智能体如何工作？

**问题：** 智能体派生子任务时，它们是共享上下文还是隔离运行？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **隔离上下文 + 仅返回摘要** | Claude Code（侧链转录稿） | 防止上下文爆炸（约 7 倍 token 开销）。但子智能体之间无法共享细粒度状态。 |
| **共享上下文** | 一些多智能体框架 | 信息完全共享。但上下文会随 N 个智能体快速填满。 |
| **消息传递** | Actor 模型系统 | 边界清晰。但需要显式设计协议。 |

**来自 Claude Code 的关键洞察：** 子智能体会话的 token 开销约为普通会话的 7 倍。只有摘要会返回给父级——完整历史永远不会进入父级的上下文。这对保护上下文至关重要。

**可以问自己的问题：**
- 你的子任务之间需要看到彼此的工作吗？
- 你如何防止 N 个子智能体消耗掉 N × context_window 个 token？
- 子智能体继承父级权限，还是自己建立权限？

## 决策 6：会话如何持久化？

**问题：** 会话结束时会发生什么？什么会延续下来？

| 方法 | 示例 | 权衡 |
|:---------|:--------|:----------|
| **仅追加的 JSONL** | Claude Code | 可审计、可重建、简单。但查询能力弱。 |
| **数据库** | 一些企业智能体 | 查询能力强、查找快。但增加基础设施依赖并降低透明度。 |
| **无状态** | 大多数聊天 API | 最简单。但没有恢复、没有分支、没有审计跟踪。 |

**来自 Claude Code 的关键洞察：** **恢复会话时绝不自动恢复权限。** 信任总是在当前会话中重新建立。安全状态不应跨会话边界隐式保留。

**关于可审计性的关键洞察：** 仅追加的 JSONL 意味着每条事件都可以被人工阅读、可纳入版本控制，无需专用工具即可重建。为此牺牲一点查询能力，是划算的。

## 元模式：三个反复出现的设计承诺

在全部六个决策中，有三种模式在 Claude Code 的架构里反复出现：

1. **渐进式分层优于单体机制** —— 安全、上下文和可扩展性都采用"堆叠独立阶段"而非单一方案。

2. **仅追加设计：可审计性优于查询能力** —— 一切皆可重建；任何已写入的数据都不会被就地改写。

3. **确定性 harness 之内的模型判断** —— 模型自由决策；harness 强制执行边界。1.6%/98.4% 这个比例并非偶然。
